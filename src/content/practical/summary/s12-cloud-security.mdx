> 섹션 12는 클라우드 서비스 모델(IaaS/PaaS/SaaS)과 배포 모델(Public/Private/Hybrid), SecaaS의 특징, CASB·SWG·ZTNA의 개념과 역할이 핵심 출제 포인트입니다. 상대적으로 분량이 짧은 섹션이지만, 최근 시험에서 클라우드 보안 브로커(CASB)와 Zero Trust 개념이 신규 출제 트렌드로 떠오르고 있습니다.

## 1. 개념 및 배경

클라우드 컴퓨팅은 IT 자원(서버, 스토리지, 소프트웨어)을 인터넷을 통해 서비스 형태로 제공하는 컴퓨팅 패러다임입니다. 기존 On-Premise 환경과 달리 공유 인프라 위에서 운영되기 때문에 가시성 부족, 멀티테넌시(Multi-tenancy) 위험, 데이터 주권 문제, 책임 공유 모델이 핵심 보안 이슈입니다.

## 2. 동작 원리 / 구조

### ☁️ 클라우드 서비스 모델 3가지 (IaaS / PaaS / SaaS)

시험 1순위 — 서비스 모델과 대표 예시 암기 필수!

| 모델 | 영문명 | 제공 범위 | 보안 책임 | 대표 서비스 |
| --- | --- | --- | --- | --- |
| IaaS | Infrastructure as a Service | 가상 서버·스토리지·네트워크 | 사용자 (OS, 미들웨어, 앱) | AWS S3, AWS EC2 |
| PaaS | Platform as a Service | 개발 플랫폼 (OS+미들웨어) | 공유 (앱, 데이터만 사용자) | MS Azure, Google AppEngine, Oracle PaaS |
| SaaS | Software as a Service | 완성된 소프트웨어 제공 | 클라우드 제공자 (대부분) | DropBox, N드라이브, Gmail |

> 📌 보안 책임 핵심: IaaS는 사용자 책임이 가장 크고, SaaS는 제공자 책임이 가장 큼. "IaaS → PaaS → SaaS 순으로 사용자 보안 책임 감소"

### 🔍 보완: 클라우드 책임 공유 모델 (Shared Responsibility Model)

제공자(CSP)와 사용자가 보안 책임을 계층별로 나누어 지는 모델로, 서비스 모델에 따라 경계가 달라집니다. 서술형에서 "각 모델에서 사용자가 책임지는 계층"을 자주 묻습니다.

| 계층(위→아래) | IaaS | PaaS | SaaS |
| --- | --- | --- | --- |
| 데이터 / 계정·접근권한 | 사용자 | 사용자 | 사용자 |
| 애플리케이션 | 사용자 | 사용자 | 제공자 |
| 미들웨어 / 런타임 | 사용자 | 제공자 | 제공자 |
| OS / 가상화 | 사용자(OS)·제공자(가상화) | 제공자 | 제공자 |
| 물리 인프라(서버·네트워크·시설) | 제공자 | 제공자 | 제공자 |

> 📌 어떤 모델이든 **데이터·계정·접근권한(IAM)** 관리는 항상 사용자 책임입니다. 물리 인프라는 항상 제공자 책임입니다. 이 두 불변 경계를 먼저 답안에 적으면 감점이 없습니다.

### 🏗️ 클라우드 배포 모델 4가지

빈칸 채우기·서술형 단골 출제!

| 배포 모델 | 설명 | 특징 |
| --- | --- | --- |
| Public Cloud | 불특정 다수에게 인터넷으로 제공 | 비용 저렴, 보안 위험 높음 |
| Private Cloud | 단일 조직 전용으로 구축·운영 | 높은 보안, 높은 비용 |
| Hybrid Cloud | Public + Private 혼합 운영 | 유연성 높음, 관리 복잡 |
| Community Cloud | 동일 목적·규제 조직 간 공유 | 금융권·공공기관 등 활용 |

### 🔐 SecaaS (Security as a Service)

클라우드 기반 보안 서비스 — SecaaS는 보안 솔루션 자체를 클라우드 서비스로 제공하는 개념입니다.

SecaaS 3대 특징

- 서비스 범위: 인증, 안티바이러스, 침입탐지, DLP, IAM 등 다양한 보안 기능 제공
- 장점: 초기 투자 비용 절감, 탄력적 확장(Scalability), 최신 위협 대응 용이
- 단점: 인터넷 의존성, 데이터 주권 문제, 공급자 서비스 중단 시 위험

### 🛡️ 클라우드 보안 핵심 기술 3종 (CASB / SWG / ZTNA)

최신 기출 트렌드 — 개념·역할 구분 출제!

| 기술 | 영문 풀네임 | 역할 | 위치 |
| --- | --- | --- | --- |
| CASB | Cloud Access Security Broker | 사용자와 클라우드(SaaS) 사이에서 가시성·컴플라이언스·데이터 보안·위협 방어 제공 | 클라우드 접근 중간 프록시 |
| SWG | Secure Web Gateway | 인터넷 트래픽을 필터링·모니터링하여 악성 사이트 접근 차단 | 웹 접근 보안 게이트웨이 |
| ZTNA | Zero Trust Network Access | "아무도 신뢰하지 않는다" 원칙 — 내부 사용자라도 매번 인증·인가 수행 | 네트워크 접근 제어 |

CASB 4대 기능 (시험 단골!)

1. 가시성 (Visibility): 클라우드 서비스 사용 현황 파악
2. 컴플라이언스 (Compliance): 규정 준수 여부 모니터링
3. 데이터 보안 (Data Security): DLP 연동, 민감 데이터 유출 방지
4. 위협 방어 (Threat Protection): 악성 행위 탐지·차단

### 🔍 보완: ZTNA vs VPN (원격 접근 보안 비교)

ZTNA는 전통적 VPN을 대체하는 제로 트러스트 기반 원격 접근 기술로, 최근 시험에서 차이점 서술이 출제됩니다.

| 구분 | VPN | ZTNA |
| --- | --- | --- |
| 신뢰 모델 | 인증되면 내부망 전체 신뢰(경계 기반) | 아무도 신뢰하지 않음, 요청마다 검증 |
| 접근 범위 | 네트워크 단위(연결 후 내부망 광범위 접근) | 애플리케이션·리소스 단위 최소 접근 |
| 접근 후 통제 | 내부 이동(lateral movement) 통제 취약 | 세션·컨텍스트별 지속 검증, 횡적 이동 차단 |
| 리소스 노출 | 게이트웨이 IP가 외부에 노출 | 리소스를 숨김(dark), 인가된 사용자만 연결 |

> 📌 핵심 정의: ZTNA(Zero Trust Network Access)는 "사용자·기기·컨텍스트를 매 요청마다 검증하고, 인가된 특정 애플리케이션에만 최소 권한으로 접근을 허용"하는 모델입니다. VPN이 '연결하면 신뢰'라면 ZTNA는 '검증해야 접근'입니다.

### 📱 모바일 보안 (MDM / MAM / MCM / EMM)

클라우드·모바일 환경 연계 출제 빈도 높음

| 기술 | 영문명 | 역할 |
| --- | --- | --- |
| MDM | Mobile Device Management | 기기 전체 제어 (원격 잠금·삭제·정책 배포) |
| MAM | Mobile Application Management | 앱 단위 제어 (업무 앱만 관리) |
| MCM | Mobile Content Management | 콘텐츠·문서 단위 보안 관리 |
| EMM | Enterprise Mobility Management | MDM + MAM + MCM 통합 솔루션 |

> 📌 BYOD 보안: 개인 소유 기기 업무 사용 → MDM으로 기기 정책 적용, MAM으로 업무 앱만 분리 관리

### 🖥️ 가상화 보안 (클라우드 기반 인프라)

클라우드의 기반 기술인 가상화(Virtualization) 관련 보안

| 구성 요소 | 보안 위협 | 대응책 |
| --- | --- | --- |
| 하이퍼바이저(Hypervisor) | VM Escape (가상머신 탈출 공격) | 패치 관리, 하이퍼바이저 접근 통제 |
| VM(가상머신) 간 통신 | VM-to-VM 스니핑 | 가상 네트워크 세분화(VLAN) |
| 스냅샷(Snapshot) | 민감 데이터 포함 유출 | 스냅샷 암호화, 접근 제한 |
| 컨테이너(Docker 등) | 컨테이너 이탈, 이미지 취약점 | 이미지 서명 검증, 네임스페이스 격리 |

### 🔑 클라우드 IAM (Identity & Access Management)

- EAM (Enterprise Access Management): OS, DB, 서버 등 엔터프라이즈 자원에 대한 통합 접근 관리
- IAM (Identity & Access Management): EAM의 확장 개념 — 클라우드 환경에서의 신원 관리 + 접근 제어 통합

주요 기능:

1. Identity (신원 확인): 사용자 인증
2. Access (접근 제어): 권한 부여 (RBAC, ABAC)
3. Lifecycle: 계정 생성 → 수정 → 삭제 관리

## 3. 예시 / 적용 시나리오

### 기출 시나리오: 클라우드 서비스 모델 구분

```text
Q. 다음 클라우드 서비스를 IaaS/PaaS/SaaS로 분류하시오.
   - AWS EC2 (가상 서버 임대)
   - MS Azure App Service (웹 앱 개발 플랫폼)
   - Gmail, Office 365 (완성된 소프트웨어)

정답:
   - AWS EC2 → IaaS (인프라만 제공, OS부터 사용자 관리)
   - MS Azure → PaaS (개발 플랫폼 제공, 앱만 사용자 개발)
   - Gmail, Office 365 → SaaS (완성 SW, 사용자는 그냥 사용)
```

### 기출 시나리오: CASB 역할 서술

```text
Q. SaaS 클라우드 서비스 사용 증가로 데이터 유출 위험이 높아졌다.
   이를 통제하기 위한 CASB를 설명하시오.

모범 답안:
CASB(Cloud Access Security Broker)는 사용자와 클라우드 서비스
사이에 위치하여 다음 4가지 기능을 수행합니다.
1. 가시성: 클라우드 서비스 사용 현황 모니터링
2. 컴플라이언스: 규정 준수 여부 확인
3. 데이터 보안: DLP 연동을 통한 민감 정보 유출 차단
4. 위협 방어: 비정상 행위 탐지 및 차단
```

### 기출 시나리오: BYOD 보안 3가지

```text
Q. 직원 개인 기기를 업무에 활용하는 BYOD 환경의 보안 대책 3가지를 쓰시오.

1. MDM(Mobile Device Management): 기기 단위 보안 정책 적용,
   원격 잠금 및 데이터 삭제 기능
2. MAM(Mobile Application Management): 업무용 앱만 분리하여
   관리, 개인 앱과 업무 앱 컨테이너 분리
3. MCM(Mobile Content Management): 업무 문서·콘텐츠 단위
   암호화 및 접근 통제
```

## 4. 실무 및 시험 포인트 정리

| 주제 | 핵심 암기 내용 | 출제 빈도 |
| --- | --- | --- |
| 서비스 모델 3종 | IaaS(인프라), PaaS(플랫폼), SaaS(소프트웨어) + 대표 예시 암기 | ⭐⭐⭐ |
| 배포 모델 4종 | Public, Private, Hybrid, Community | ⭐⭐⭐ |
| 보안 책임 모델 | IaaS(사용자 책임 大) → SaaS(제공자 책임 大), 데이터·IAM은 항상 사용자 책임 | ⭐⭐⭐ |
| ZTNA vs VPN | VPN=경계 신뢰·네트워크 단위 / ZTNA=매 요청 검증·앱 단위 최소 접근 | ⭐⭐ |
| CASB 4기능 | 가시성, 컴플라이언스, 데이터 보안, 위협 방어 | ⭐⭐⭐ |
| SecaaS 특징 | 클라우드 기반 보안 서비스, Scalability, DLP·IAM 제공 | ⭐⭐ |
| ZTNA 개념 | Zero Trust = "내부도 신뢰 안 함", 매번 인증·인가 | ⭐⭐ |
| SWG 개념 | 웹 트래픽 필터링·모니터링 게이트웨이 | ⭐⭐ |
| MDM/MAM/EMM | MDM(기기), MAM(앱), MCM(콘텐츠), EMM(통합) | ⭐⭐ |
| 가상화 보안 | VM Escape, 하이퍼바이저 보안, 컨테이너 이미지 취약점 | ⭐⭐ |
| IAM vs EAM | EAM(엔터프라이즈 자원 접근), IAM(신원+접근 통합 관리) | ⭐⭐ |

> 📌 서술형 고득점 전략: 클라우드 보안 문제는 서비스 모델(IaaS/PaaS/SaaS) + 배포 모델 + CASB 4기능을 한 세트로 묶어 외우세요. CASB 문제는 반드시 4가지 기능(가시성·컴플라이언스·데이터보안·위협방어)을 나열하고, ZTNA는 "Zero Trust = 내부 사용자도 신뢰하지 않는 접근 통제 모델"로 한 줄 정의를 작성하면 고득점입니다.
